Sistema SUNAT - Facturin

Facturin implementa la integración completa con SUNAT para la facturación electrónica peruana siguiendo el estándar UBL 2.1. Este documento describe la arquitectura, flujo de datos y componentes del sistema.

Versión: 1.0.0 | Estándar: UBL 2.1 | Entornos: Beta y Producción

1. Arquitectura General

El sistema SUNAT de Facturin sigue una arquitectura de procesamiento asíncrono para nunca bloquear las solicitudes HTTP:

Cliente crea comprobante via API
Servicio guarda en BD y encola job
Worker genera XML UBL 2.1
Worker firma XML con certificado digital
Worker envía a SUNAT (sendBill)
SUNAT devuelve ticket (async) o CDR (sync)
Si async: encola job consultar-cdr
Worker consulta estado (getStatus)
SUNAT devuelve CDR final
Worker actualiza estado en BD

5. Operaciones SOAP

5.1 sendBill (Envío de Documento)

Envía un comprobante firmado a SUNAT para procesamiento.

<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
  <soapenv:Header>
    <wsse:Security>
      <wsse:UsernameToken>
        <wsse:Username>USUARIO</wsse:Username>
        <wsse:Password>CONTRASEÑA</wsse:Password>
      </wsse:UsernameToken>
    </wsse:Security>
  </soapenv:Header>
  <soapenv:Body>
    <ser:sendBill>
      <fileName>20480001234-01-F001-00000001.zip</fileName>
      <contentFile>BASE64_XML_CONTENT</contentFile>
    </ser:sendBill>
  </soapenv:Body>
</soapenv:Envelope>

5.2 getStatus (Consulta de Ticket)

Consulta el estado de un ticket obtenido previamente.

<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
  <soapenv:Header>
    <wsse:Security>
      <wsse:UsernameToken>
        <wsse:Username>USUARIO</wsse:Username>
        <wsse:Password>CONTRASEÑA</wsse:Password>
      </wsse:UsernameToken>
    </wsse:Security>
  </soapenv:Header>
  <soapenv:Body>
    <ser:getStatus>
      <ticket>NUMERO_TICKET</ticket>
    </ser:getStatus>
  </soapenv:Body>
</soapenv:Envelope>

6. Estados del Comprobante

El campo sunatEstado en la tabla comprobantes sigue esta máquina de estados:

pendiente - Creado, no enviado
enviado - Ticket recibido, esperando CDR
aceptado - CDR recibido con status 0
rechazado - CDR con código 2000-3999
excepcion - Error o código 99
anulado - Dado de baja
en_proceso - En procesamiento

7. Códigos de Estado SUNAT

Código Estado Significado
0 ACEPTADO Documento aceptado por SUNAT
98 EN_PROCESO SUNAT está procesando el documento
99 EXCEPCION Error en el procesamiento
2000-2999 RECHAZADO Documento rechazado (error del emisor)
3000-3999 RECHAZADO Documento rechazado (error del receptor)
4000+ ACEPTADO Documento aceptado con observaciones

8. Códigos de Error Comunes

Código Descripción Causa
1000 Error de autenticación Usuario o contraseña SUNAT incorrectos
1001 Acceso denegado Cuenta no tiene permisos para el servicio
1002 Error de certificado Certificado digital inválido o expirado
1007 Timeout SUNAT no respondió a tiempo
PARSE_ERROR Error de parsing No se pudo interpretar la respuesta SUNAT
SYSTEM_ERROR Error del sistema Error HTTP o de conexión

9. Flujo de Datos - Detalle

9.1 Creación de Comprobante

  1. Cliente envía POST a /v1/comprobantes con los datos del comprobante
  2. El servicio valida los datos (serie, número, RUC cliente, IGV)
  3. Se guarda en BD con estado pendiente
  4. Se encola job sunat-send con {comprobanteId, tenantId}
  5. Respuesta inmediata al cliente con el comprobante creado

9.2 Procesamiento del Job enviar-comprobante

  1. Worker obtiene el job de la cola sunat-send
  2. Recupera comprobante y tenant de la BD
  3. Valida que el tenant tenga certificado y credenciales SUNAT
  4. Genera XML UBL 2.1 con generateXML()
  5. Firma el XML con signXML() usando el certificado digital
  6. Crea cliente SUNAT y llama a sendBill()
  7. Parsea la respuesta:
    • Con ticket: Almacena ticket, actualiza estado a enviado, encola consultar-cdr
    • CDR directo: Actualiza estado según código de respuesta
  8. Registra resultado en logs

9.3 Procesamiento del Job consultar-cdr

  1. Worker obtiene job con {comprobanteId, tenantId, ticket}
  2. Crea cliente SUNAT y llama a getStatus(ticket)
  3. Si statusCode === 0: Actualiza estado a aceptado, guarda CDR
  4. Si statusCode === 98: Re-encola el job con backoff exponencial (max 20 intentos)
  5. Si statusCode >= 2000: Actualiza estado a rechazado
  6. Si statusCode === 99: Actualiza estado a excepcion

11. Esquema de Base de Datos

Tabla: tenants

Credenciales y certificados por tenant:

Campo Tipo Descripción
certificadoDigital text Certificado .p12/.pfx en Base64
certificadoPassword text Contraseña del certificado
sunatUsername text Usuario SOL SUNAT
sunatPassword text Contraseña SOL SUNAT

Tabla: comprobantes

Datos del comprobante y estado SUNAT:

Campo Tipo Descripción
xmlContent text XML UBL generado
cdrContent text Respuesta CDR de SUNAT
cdrStatus text Estado CDR (ACEPTADO, RECHAZADO, etc)
sunatTicket varchar Número de ticket SUNAT
sunatEstado text Estado del proceso SUNAT
sunatFechaEnvio timestamp Fecha de envío a SUNAT
sunatFechaRespuesta timestamp Fecha de respuesta CDR

Tabla: sunat_logs

Logs de comunicación con SUNAT (actualmente no implementado):

Campo Tipo Descripción
tenantId uuid ID del tenant
comprobanteId uuid ID del comprobante
tipoOperacion text sendBill o getStatus
ticket varchar Número de ticket
requestXml text XML enviado
responseXml text Respuesta XML
errorCode text Código de error
errorMessage text Mensaje de error
Nota de Seguridad: Los campos certificadoPassword y sunatPassword se almacenan sin encriptar. Según las prácticas de seguridad de Facturin, estos campos deberían estar encriptados en la base de datos.

13. Consideraciones y Limitaciones

Implementado

  • Generación de XML UBL 2.1 para facturas, boletas, notas de crédito y débito
  • Firma digital con certificados .p12/.pfx
  • Envío asíncrono via SOAP (sendBill)
  • Consulta de estado via SOAP (getStatus)
  • Procesamiento de CDR (Constancia de Recepción)
  • Manejo de errores con códigos SUNAT
  • Multi-tenancy con credenciales por tenant

No Implementado / Parcial

  • Guía de Remisión (09) - en catálogos pero no en generator
  • Comprobante de Percepción (40) - en catálogos pero no en generator
  • Resumen Diario (RC) - no implementado
  • Comunicación de Baja (RA) - no implementado
  • Extracción del ZIP del CDR - el ZIP se guarda sin descomprimir
  • Logging a tabla sunat_logs - esquema existe pero no se usa
  • Validación XSD del XML antes de firmar
  • Ambiente SUNAT por tenant - global de env var
